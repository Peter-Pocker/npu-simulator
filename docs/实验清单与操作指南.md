# 论文实验清单：目的、逻辑、展示内容与操作步骤

本文档按第五章结构梳理你需要做的全部实验，并给出每一步的具体做法。

---

## 一、实验总览与逻辑关系

| 实验块 | 目的 | 逻辑位置 | 论文中展示 |
|--------|------|----------|------------|
| **5.1 微基准（MB-1, MB-2）** | 验证仿真器**正确性**：依赖因果、核间/DRAM闭环、simple vs full 差异 | 先证明“跑得对” | 图5-1～5-3 |
| **5.1 ZeBu 对比** | 用**硬件仿真**佐证周期预测**精度** | 再证明“周期准” | 表 + 图5-3a |
| **5.2 真实负载瓶颈分析** | 用调度器 IR 做**可解释瓶颈归因** | 证明仿真器能支撑分析 | 图5-4～5-6 |
| **5.3 微架构 DSE** | 展示**设计空间探索**价值：边际收益、瓶颈迁移、最优点差异 | 证明仿真器能支撑 DSE | 图5-7～5-10 |

**整体逻辑**：微基准证明正确性 → ZeBu 证明精度 → 真实负载证明可解释性 → DSE 证明探索价值。

## 二、各实验的目的、逻辑与论文展示

### 实验 1：MB-1 双核点对点依赖

- **目的**：验证（1）Core1 在收到 Core0 的数据前不会进入 COMPUTING；（2）核间 transfer 依赖在 trace 中可观测；（3）仿真能稳定结束。
- **逻辑**：若 MB-1 通过，说明“依赖驱动 + NoC 投递”的因果顺序正确，是后续所有实验的基础。
- **论文中展示**：
  - **图5-1**：MB-1 的**状态转移时间线**（横轴 cycle，纵轴 core，颜色区分 LOADING/COMPUTING/WRITEBACK）。来源：仿真器带 `-t` 导出的 `state_trace.csv`，用脚本或 Excel 画时间线/甘特图。
  - **图5-2**：两核的**阶段占比**（Loading / Compute / Writeback 的周期数或占比饼图/柱状图）。来源：`print_stats()` 的 per-core 统计或 `workload_summary.csv` 聚合。

**你需要**：一份 MB-1 的 IR（双核，Core0 产出某 transfer，Core1 依赖该 transfer）。若无现成 IR，需用调度器或手写 JSON 构造最小双核依赖用例；然后用该 IR 跑仿真并开 `-t` 导出 trace。

---

### 实验 2：MB-2 仅 DRAM 依赖

- **目的**：验证（1）无核间通信时仿真仍闭环；（2）**simple** 模式近似固定延迟；（3）**full** 模式在并发度增加时总周期上升（排队/拥塞）。
- **逻辑**：说明 NoC/DRAM 两档后端的差异是可观测的，为 5.2 的瓶颈分解和 5.3 的 DSE 提供依据。
- **论文中展示**：
  - **图5-3**：**并发度 vs 总周期** 曲线，两条线分别对应 simple 与 full。横轴为并发度（如 DRAM 请求并发数或 workload 数），纵轴为总周期。

**你需要**：仅 DRAM 依赖的 IR（多 workload 均从 DRAM 读、写回 DRAM）；同一 IR 用两种 config（simple NoC/DRAM 与 full BookSim2/Ramulator2）各跑若干遍，记录总周期，画曲线。

---

### 实验 3：与 ZeBu 硬件仿真的对比（单核）

- **目的**：用 ZeBu 单核 trace 作为**金标准**，验证仿真器在“同一调度 IR”下的总周期预测是否接近。
- **逻辑**：ZeBu 是硬件仿真，周期来自真实 RTL/行为；若仿真器与 ZeBu 在 ±5% 内，可认为时序模型可信，后续瓶颈分析与 DSE 的结论才有依据。
- **论文中展示**：
  - **表（表\ref{tab:zebu_validation}）**：模型、批大小、ZeBu 周期、仿真器周期、相对误差（%）。用 `scripts/batch_compare_zebu.py` 生成。
  - **图5-3a**（可选）：ZeBu vs 仿真器周期的柱状图或散点图（y=x 附近 ±5% 带）。

**操作步骤**见下文「三、具体操作步骤」中的 ZeBu 部分。

---

### 实验 4：真实 AI 负载瓶颈分析（ResNet34/50）

- **目的**：用调度器生成的 ResNet34/ResNet50 IR 驱动仿真，对端到端周期做 **Compute / NoC_Stall / Mem_Stall** 分解，并找出热点核与拖尾原因。
- **逻辑**：展示仿真器不仅能给出总周期，还能给出“时间花在哪”，为架构优化提供依据。
- **论文中展示**：
  - **图5-4**：端到端**瓶颈占比**（饼图或堆叠柱状图：Compute、NoC_Stall、Mem_Stall）。数据来自各核的 `cycles_computing`、`cycles_stall_noc`、load/wb 相关统计。
  - **图5-5**：**热点核热力图**（横轴 core_id，纵轴阶段或时间占比，颜色表示负载）。
  - **图5-6**：**代表性核心的 workload 甘特图**（横轴 cycle，纵轴 workload 或阶段）。数据来自 `trace/workload_summary.csv`，可用 `scripts/plot_gantt.py` 或类似脚本画。

**你需要**：ResNet34/ResNet50 的 stschedule.json（如 Gemini-Compiler-IR 的 scheduler_output），用同一 config 跑仿真并开 `-t`，再从 stats 与 trace 中算占比、画图。

---

### 实验 5：微架构 DSE（mac_units × num_vcs）

- **目的**：证明（1）增加计算资源（mac_units）有**边际收益递减**且伴随**瓶颈迁移**；（2）**不同负载（ResNet34 vs ResNet50）的最优配置点不同**。
- **逻辑**：说明仿真器能支撑“改参数→看周期→找最优点”的探索流程，体现设计空间探索价值。
- **论文中展示**：
  - **图5-7**：总周期 **vs mac_units**（两条曲线：ResNet34、ResNet50），可选 2～4 个 mac_units 取值。
  - **图5-8**：总周期 **vs num_vcs**（两条曲线：ResNet34、ResNet50），可选 2～4 个 num_vcs 取值。
  - **图5-9**：ResNet34 在 **mac_units × num_vcs** 网格上的热力图，标注最优点。
  - **图5-10**：ResNet50 同上。

**你需要**：固定 IR（如 ResNet34 与 ResNet50 各一份），在 config 中扫 mac_units（如 64, 128, 256, 512）和 num_vcs（如 2, 4, 8, 16），每个组合跑一次仿真记录总周期，填表并画热力图与曲线。

## 三、具体操作步骤（怎么做这些实验）

### 前置：编译仿真器与准备配置

```bash
# 在项目根目录
mkdir -p build && cd build
cmake .. && make
# 得到可执行文件 npu_sim（可能在 build/ 或当前目录）
```

准备一份基础 `config.json`（若没有可从 `include/npu_sim/config.h` 或现有示例抄写），包含 `core`、`sram`、`noc`、`dram`、`topology` 等字段；DSE 时只改其中 `core.mac_units` 和 NoC 的 `num_vcs`（具体键名以你代码为准）。

---

### 步骤 A：MB-1 双核点对点依赖

1. **准备 IR**：需要 2 核、Core0 有一个 workload 输出到 Core1、Core1 有一个 workload 依赖该 transfer。若项目中有 `trans_2x2_ir.json` 等双核用例，可先检查是否满足“Core0→Core1 依赖”；否则需手写或由调度器生成最小 JSON。
2. **运行仿真并导出 trace**：
   ```bash
   ./npu_sim -i /path/to/mb1_two_core_p2p.json -c config.json -t trace_mb1
   ```
3. **出图**：
   - 图5-1：用 `trace_mb1/state_trace.csv`，按 `(cycle, core_id, new_state)` 画时间线（横轴 cycle，纵轴 core，状态用颜色区分）。
   - 图5-2：用 `print_stats()` 输出的各核 Idle/Loading/Compute/Writeback 周期，或从 `trace_mb1/workload_summary.csv` 聚合，画柱状图/饼图。

---

### 步骤 B：MB-2 仅 DRAM 依赖

1. **准备 IR**：所有 workload 的输入仅来自 DRAM、输出仅写回 DRAM（无 core-to-core）。可手写或由调度器生成。
2. **准备两套配置**：`config_simple.json`（NoC/DRAM 为 simple）与 `config_full.json`（BookSim2 + Ramulator2）。
3. **扫并发度**：例如 1、2、4、8 个 workload（或通过多核/多 batch 实现不同并发），对每个并发度分别用 simple 和 full 各跑一次，记录 Total cycles。
4. **出图**：图5-3 横轴为并发度，纵轴为总周期，两条线分别为 simple 与 full。

---

### 步骤 C：ZeBu 对比（单核）

1. **ZeBu 周期**（已有数据）：  
   ```bash
   python3 scripts/parse_zebu_trace.py third_party/Gemini-Compiler-IR/zebu_trace_output/int8_resnet50.sim_quantized_b64_c1_bw16_sim.txt
   ```  
   或直接用 `scripts/batch_compare_zebu.py` 会自动读 Zebu 目录下所有 `*_sim.txt` 得到 ZeBu 周期。

2. **仿真器周期**：对每个配置（ResNet34/50 × batch 1/4/16/64）用**同一份** stschedule 和**单核** config 跑：
   ```bash
   ./npu_sim -i third_party/Gemini-Compiler-IR/scheduler_output/int8_resnet50.sim_quantized_b64_c1_bw16_stschedule.json -c config.json 2>&1 | tee sim_b64.log
   ```
   在 log 中记下 “Total cycles: N”。

3. **填表与 LaTeX**：把各配置的仿真器周期写入 `sim_results.csv`（表头：`model,batch,sim_cycles`），运行：
   ```bash
   python3 scripts/batch_compare_zebu.py --zebu-dir third_party/Gemini-Compiler-IR/zebu_trace_output --sim-results sim_results.csv --latex --csv-out zebu_validation_table.csv
   ```
   将输出的 LaTeX 表格粘贴到 c5.tex 中替换表\ref{tab:zebu_validation} 的“（待填）”行。

4. **图5-3a（可选）**：用 `zebu_validation_table.csv` 或脚本输出画柱状图（ZeBu vs 仿真器）或散点图（仿真器周期 vs ZeBu 周期，y=x 与 ±5% 带）。

---

### 步骤 D：真实负载瓶颈分析（图5-4～5-6）

1. **跑仿真并导出 trace**（ResNet34 与 ResNet50 各至少一份）：
   ```bash
   ./npu_sim -i third_party/Gemini-Compiler-IR/scheduler_output/int8_resnet50.sim_quantized_b16_c1_bw16_stschedule.json -c config.json -t trace_rn50
   ```

2. **图5-4 瓶颈占比**：从 `print_stats()` 或代码中的 per-core 统计得到各核的 `cycles_computing`、`cycles_loading`、`cycles_writeback`、`cycles_stall_noc`，按论文公式聚合成 Compute、NoC_Stall、Mem_Stall，画饼图或堆叠柱状图。

3. **图5-5 热点核热力图**：以 core_id 为横轴，阶段（或 Load/Compute/WB 占比）为纵轴或颜色，用热力图表示每核负载。

4. **图5-6 workload 甘特图**：用 `trace_rn50/workload_summary.csv` 和 `scripts/plot_gantt.py`：
   ```bash
   python3 scripts/plot_gantt.py --trace trace_rn50/state_trace.csv --output fig_c5_gantt.png
   ```
   若脚本输出是 core 状态甘特图，可再根据 `workload_summary.csv` 画“某核的 workload 时间线”作为补充。

---

### 步骤 E：微架构 DSE（图5-7～5-10）

1. **定 IR**：固定 ResNet34 与 ResNet50 各一个 stschedule（如 b16 单核）。
2. **扫 mac_units**：在 config 中设 `mac_units = 64, 128, 256, 512`（或你代码中的键名），对 ResNet34 与 ResNet50 各跑一遍，记录总周期 → 画图5-7。
3. **扫 num_vcs**：在 config 中设 NoC 的 `num_vcs = 2, 4, 8, 16`，同样各跑一遍 → 画图5-8。
4. **二维网格**：对 (mac_units × num_vcs) 做 4×4 共 16 个点，ResNet34 与 ResNet50 各跑 16 次，记录总周期，填成矩阵画热力图，并标出最优点 → 图5-9、图5-10。

可用 shell 或 Python 循环批量改 config、调 `npu_sim`、从 stdout 解析 Total cycles 并写入 CSV，再画图。

## 五、建议顺序

1. **ZeBu 对比**（步骤 C）：数据现成、脚本齐全，先填表和图5-3a，快速形成“精度验证”结论。  
2. **真实负载 + trace**（步骤 D）：跑 ResNet34/50 并开 `-t`，出图5-4～5-6，形成“瓶颈分析”一节。  
3. **DSE**（步骤 E）：扫参数、画图5-7～5-10，形成“设计空间探索”一节。  
4. **微基准**（步骤 A、B）：若有现成 MB-1/MB-2 IR 则跑并出图5-1～5-3；若无则需先构造 IR 或简化描述（如只做 ZeBu + 真实负载 + DSE 也可支撑贡献）。

按上述顺序做，可以在论文中先有“精度—瓶颈—探索”的完整证据链，再补微基准细节。
